# Streamlit Dashboard Engineering Documentation

## 1. Purpose

This document explains the full engineering plan for building and deploying the public dashboard for the Last Mile Logistics Auditor project using Python and Streamlit Cloud. It is documentation only. It does not include implementation code.

The goal is to satisfy the exam requirement that the final dashboard must be available through a public link. Since an individual Power BI account may not support public web publishing without a Pro or organizational setup, Streamlit Cloud is the safest option. Streamlit Cloud can deploy a Python dashboard from a GitHub repository and provide a public URL that reviewers can open without signing in.

## 2. Why Streamlit Cloud Is the Recommended Choice

The assignment allows dashboarding tools such as Tableau Public, Looker Studio, Streamlit Cloud, Power BI Web, or similar public dashboard tools. Streamlit Cloud is a strong fit for this project because the analysis is already Python-based and the dashboard can use the small summary CSV files generated by the notebook.

Advantages:

- It creates a public link that can be added to the README.
- It does not require Power BI Pro.
- It works well with GitHub.
- It uses Python, which matches the data science workflow.
- It can read the existing `outputs/*.csv` files instead of requiring the raw Kaggle dataset.
- It avoids uploading the large raw dataset to GitHub.

The dashboard should be built from aggregated output files, not from the raw Olist CSV files. This keeps deployment light, fast, and submission-friendly.

## 3. Dashboard Objective

The Streamlit dashboard should answer the CEO's business question:

> Are delivery failures concentrated in specific regions, or is this a nationwide problem?

It should also show whether delivery performance is connected to customer sentiment.

The dashboard must support the required user stories:

1. Show delivery performance by customer state.
2. Show on-time, late, and super-late delivery outcomes.
3. Compare delivery delay with average review score.
4. Include the Candidate's Choice product category risk analysis.

The dashboard should be simple, professional, and focused. It should not look like a notebook copied into a web page. It should feel like an audit tool for a logistics leader.

## 4. Data Inputs

The dashboard should use the summary CSV files already generated by the notebook.

Required dashboard input files:

| File | Purpose |

The app should not depend on `archive/`. The raw dataset is large and should not be committed to GitHub. Streamlit should only need the small summary files in `outputs/`.

## 6. Required Dependency Updates

The current `requirements.txt` must include the dashboard libraries before deployment.

Required packages:

```text
streamlit
pandas
plotly
matplotlib
seaborn
numpy
```

If the dashboard uses only Streamlit charts and pandas, Plotly may not be strictly necessary. However, Plotly is recommended because it creates clean interactive charts with tooltips.

The final deployed app should not require Jupyter to run. Jupyter is useful for the notebook, but the Streamlit app itself should only need dashboard dependencies.

## 7. Dashboard Page Layout

The dashboard should use a single-page layout. A single-page dashboard is better for the exam because reviewers can quickly see the full story without navigating many pages.

Recommended page order:

1. Header and short business context.
2. KPI cards.
3. Geographic delivery performance.
4. Sentiment impact.
5. Delay bucket trend.
6. Candidate's Choice product category risk.
7. Business recommendations.

The page should avoid long explanation blocks. The README and presentation can contain detailed explanation. The dashboard should be visual and fast to read.

## 8. Header Section

The dashboard title should be:

```text
Last Mile Logistics Auditor
```

Subtitle:

```text
Delivery promise accuracy and customer sentiment audit for Veridi Logistics.
```

Short context:

```text
This dashboard compares estimated and actual delivery performance across regions, review scores, and product categories.
```

The header should make the business purpose clear immediately.

## 9. KPI Cards

The top of the dashboard should contain KPI cards. These cards should be calculated from the summary CSVs.

Recommended KPI cards:

| KPI | Source | Meaning |
| --- | --- | --- |
| Total Delivered Orders | Sum of `delivered_orders` from state summary | Size of analyzed delivery population |
| National Late Rate | Sum late orders / sum delivered orders | Overall delivery promise failure |
| Worst State | Highest `late_rate` state | Region needing urgent attention |
| On-Time Review Score | Sentiment summary | Customer satisfaction when delivery promise is met |
| Super-Late Review Score | Sentiment summary | Customer satisfaction when delivery promise fails badly |

KPI values should be rounded clearly:

- Percentages: one decimal place.
- Review scores: two decimals.
- Counts: comma-separated whole numbers.

## 10. Geographic Performance Section

This section answers the regional director user story.

Main question:

```text
Which states have the highest percentage of late deliveries?
```

Data source:

```text
outputs/state_delivery_summary.csv
```

Recommended visual:

- Horizontal or vertical bar chart sorted by `late_rate` descending.

Required fields:

- `customer_state`
- `late_rate`
- `delivered_orders`
- `avg_review_score`
- `super_late_rate`

Recommended interaction:

- State filter or multiselect.
- Option to show top 10, top 15, or all states.

Recommended interpretation text:

```text
Late delivery rates are ranked by customer state. Higher values indicate regions where delivery promises are missed more often.
```

This section should not overclaim that remote geography causes delays unless distance is calculated. It can say that the pattern helps identify where operational investigation should begin.

## 11. Sentiment Impact Section

This section answers the customer success user story.

Main question:

```text
Do late deliveries receive worse review scores?
```

Data source:

```text
outputs/sentiment_by_delivery_status.csv
```

Recommended visual:

- Bar chart showing average review score by `Delivery_Status`.

Required fields:

- `Delivery_Status`
- `avg_review_score`
- `order_count`

Recommended order:

1. On Time
2. Late
3. Super Late

The y-axis should use a 0-to-5 or 1-to-5 scale so the review score context is clear.

Recommended interpretation text:

```text
Average review score declines as delivery performance worsens, showing that missed delivery promises are associated with lower customer satisfaction.
```

Use “associated with” rather than “caused by” because the project is observational.

## 12. Delay Bucket Section

This section gives a more detailed view of delay severity.

Main question:

```text
How does review score change as delay days increase?
```

Data source:

```text
outputs/delay_bucket_review_summary.csv
```

Recommended visual:

- Line chart or bar chart of `avg_review_score` by `delay_bucket`.

Required fields:

- `delay_bucket`
- `avg_review_score`
- `order_count`

Recommended interpretation:

```text
This trend shows whether customer sentiment drops gradually or sharply as delays become longer.
```

This chart helps prove that super-late orders are not just operationally bad, but also damaging to customer experience.

## 13. Candidate's Choice Section

This section presents the extra business-value analysis.

Main question:

```text
Which product categories carry the highest delivery risk?
```

Data source:

```text
outputs/category_delivery_summary.csv
```

Recommended visual:

- Horizontal bar chart of top product categories by `late_rate`.

Required fields:

- `category`
- `delivered_orders`
- `late_orders`
- `late_rate`
- `avg_review_score`

Recommended filters:

- Minimum delivered orders threshold.
- Top N categories.

Recommended interpretation:

```text
Category risk analysis helps identify whether delivery problems are linked to product type, not only customer geography.
```

This section should explain why the Candidate's Choice matters. Some products may be harder to ship because they are bulky, fragile, high-value, or dependent on sellers with different fulfillment quality.

## 14. Sidebar Filters

The dashboard can include sidebar controls.

Recommended filters:

| Filter | Purpose |
| --- | --- |
| Top N states | Let viewer focus on worst regions |
| Top N categories | Let viewer focus on highest-risk product groups |
| Minimum category orders | Avoid misleading category rankings from small samples |
| Chart theme option | Optional, not required |

Filters should improve clarity. They should not make the dashboard complicated.

## 15. Business Recommendation Section

At the bottom of the dashboard, include short recommendations.

Recommended recommendation themes:

1. Prioritize operational review in states with the highest late rates.
2. Recalibrate estimated delivery dates in high-risk regions.
3. Track super-late deliveries separately because they strongly damage review scores.
4. Investigate high-risk product categories for fulfillment, packaging, or carrier issues.
5. Monitor delivery promise accuracy as an ongoing KPI.

Keep this section brief. The full explanation can go in the presentation.

## 16. Visual Design Guidelines

The dashboard should look professional and clean.

Recommended style:

- Use clear chart titles.
- Use short metric labels.
- Format percentages properly.
- Use consistent colors for delivery status.
- Avoid too many colors.
- Put the most important metrics at the top.
- Avoid large paragraphs.

Suggested delivery status colors:

| Status | Color Idea |
| --- | --- |
| On Time | Green |
| Late | Orange |
| Super Late | Red |

Avoid visual clutter. The dashboard should be easy for a recruiter or hiring manager to understand in less than two minutes.

## 17. Deployment Plan

Deployment will use Streamlit Cloud.

Steps:

1. Create or confirm a GitHub repository.
2. Make sure the repository includes `app.py`.
3. Make sure the repository includes the `outputs/` summary CSV files.
4. Make sure the repository includes `requirements.txt`.
5. Make sure the repository does not include `archive/`.
6. Go to Streamlit Cloud.
7. Connect the GitHub account.
8. Select the project repository.
9. Set the main file path to `app.py`.
10. Deploy the app.
11. Copy the public Streamlit URL.
12. Open the URL in Incognito Mode.
13. Add the URL to `README.md`.

The public URL will usually look like:

```text
https://your-app-name.streamlit.app
```

## 18. GitHub Requirements

Before deployment, GitHub should contain only the files needed for review and deployment.

## 20. Public Access Testing

The assignment is strict about public links. After deployment:

1. Copy the Streamlit link.
2. Open a private or Incognito browser window.
3. Paste the link.
4. Confirm the dashboard loads without asking for login.
5. Confirm charts are visible.
6. Confirm there are no missing-file errors.
7. Confirm the dashboard does not require the raw `archive/` folder.

If the app fails in Streamlit Cloud but works locally, the most likely causes are:

- Missing package in `requirements.txt`.
- File path case mismatch.
- Output CSV files not committed.
- App expects `archive/` instead of `outputs/`.

## 21. Local Testing Plan

Expected local command:
